# 广东推倒胡算牌器——视觉、安全与前端交互需求规约

## 1. 采集安全需求 (针对原 DirectX 方案修正)

### 1.1 背景与技术痛点
原定“通过 DirectX 锁定腾讯麻将软件画面”的技术路线具有高侵入性。腾讯游戏反作弊系统（如 ACE）对内存注入、底层渲染管线 Hook（如 Present/EndScene 函数 Hook）有极高敏感度，极易引发玩家账号被封禁或软件被拦截。

### 1.2 替代技术路线需求
系统必须采用**完全免注入、非侵入式**的画面采集方式。研发需从以下两个方案中二选一进行实现：


* **方案 A：基于 OBS 虚拟相机（Virtual Camera）的旁路读取**
    * 算牌器不直接采集游戏窗口，而是提供标准的摄像头读取接口（如 OpenCV `VideoCapture`）。
    * 由用户自行打开 OBS 软件，利用 OBS 内置的“游戏采集”或“窗口采集”抓取麻将画面，并开启“虚拟相机”向外广播。
    * **核心要求**：算牌器仅作为视频流接收方，将作弊风险降为零。

## 2. 动态自适应 UI 识别需求 (针对手牌与副露区域变动)

### 2.1 背景与技术痛点
游戏界面采用自适应动态布局。玩家每发生一次“碰”或“杠”操作，右侧的副露（落地牌）区域就会向左延伸，导致原本的手牌区域整体向右缩短、变窄。传统的固定坐标（绝对 ROI 裁剪）无法应对这种动态变化。

### 2.2 技术实现需求
系统必须能够动态识别手牌与副露。推荐采用以下两层演化策略：

#### 策略一：基于牌体三维特征的降维打击（推荐优先实现）
* **高矮特征判定**：手牌贴图资产与副露贴图资产在游戏引擎中具有不同的缩放比例。程序在切出单张牌后，需通过像素高度（Height）进行分类（如：高度大于某阈值为手牌，小于该阈值为副露）。
* **宽高比与朝向判定**：
    * 手牌牌体呈垂直矩形（宽高比约为 $1 : 1.3$）。
    * 副露中被碰牌（横放状态）呈水平矩形（宽高比约为 $1.3 : 1$）。
    * 通过牌体自身的几何特征直接反推其属于“手牌”或“副露”，不依赖其所在的屏幕坐标。

#### 策略二：动态边界分割（滑块检测）
* 系统需锁定底部持牌大条区域，沿 X 轴从左往右进行轮廓扫描。
* 利用手牌（高、窄、垂直）与副露（矮、宽、横放）之间的视觉断层空隙，动态计算出当前的**分割点坐标（$X_{\text{split}}$）**。
* **业务逻辑定义**：分割点 $X_{\text{split}}$ 左侧的所有牌体归流为 `hand_tiles`（手牌数据）；右侧所有牌体归流为 `my_melds`（副露数据）。

## 3. 基于 YOLO11 的轻量化物体检测需求

为同时解决“牌面内容识别”与“手牌/副露状态区分”两大问题，系统引入 YOLO11 进行一站式端到端检测。

### 3.1 数据集与标签设计需求
在数据集标注阶段，必须对**手牌状态**和**副露状态**的同一种牌面进行**解耦标注**，标签设计共包含 $34 \times 2 = 68$ 类：

* **手牌类标签（示例）**：`hand_1m` (手牌一万), `hand_5p` (手牌五筒), `hand_9s` (手牌九条) ...
* **副露类标签（示例）**：`meld_1m` (副露一万), `meld_5p` (副露五筒), `meld_9s` (副露九条) ...

### 3.2 模型部署与推理延迟需求
* **推理速度**：在算牌器端，训练完成的 `.pt` 模型（推荐 `yolo11n.pt` 骨干网络）必须导出为 **ONNX 格式**。
* **运行效率**：利用 ONNX Runtime 或 OpenCV `DNN` 模块在 CPU 上进行单帧推理，单帧延迟必须控制在 **20ms 以内**，确保算牌器决策的实时响应。

## 4. 核心算牌算法逻辑修正

### 4.1 纠正 13 张牌切牌相公错误
* **严格约束**：AI 算牌器的切牌决策层必须遵循**“14张牌（摸牌后/碰牌后）才进行切牌推荐”**的硬性铁律。禁止在 13 张牌（等待状态）时提示玩家打牌，防止造成死牌（相公）。

### 4.2 确立“速度第一”的推倒胡决策原则
* **番型权重修正**：广东推倒胡规则由于自摸收益极高，且没有繁琐的起胡番数限制。算法层必须修改“为了强追碰碰胡而强行拆顺子退向听”的机械化加权逻辑。
* **引入期望收益（EV）模型**：放弃纯粹的静态线性加权，改用期望收益模型计算出牌路线：
  $$\text{期望收益 (EV)} = \text{成功和牌率} \times \text{该路线和牌得分（含买马、杠得分期望）}$$
* **战略核心**：当手牌已经听牌或面临高速度路线时，**速度权重必须处于最高优先级**，杜绝“降智”拆牌行为。

## 5. 桌面悬浮窗（Overlay）前端交互需求

### 5.1 功能概述
本模块负责将 AI 决策核心输出的「出牌推荐」、「碰/杠提示」及「牌效分数」，以**不干扰游戏操作、不遮挡视线、极致直观**的视觉形式实时反馈给玩家。系统采用桌面透明置顶悬浮窗（Overlay）技术，将提示信息精准叠加在腾讯麻将游戏窗口之上。

### 5.2 技术实现与性能需求
* **视窗属性定义**（以 C++ Qt / C# WinForm 或 Electron 实现）：
    * **置顶显示（Always on Top）**：必须具备 `WS_EX_TOPMOST` 属性，确保永远层叠在麻将游戏窗口上方，不被游戏画面覆盖。
    * **完全透明（Transparent Background）**：视窗背景必须为 100% 透明（Alpha = 0），仅渲染提示组件本身。
    * **鼠标穿透（Click-Through）**：必须具备 `WS_EX_LAYERED` 与 `WS_EX_TRANSPARENT` 属性。玩家在悬浮窗区域内点击鼠标时，点击事件必须直接穿透传递给下方的麻将游戏，**绝不可阻挡玩家正常的搓牌、出牌操作**。
* **窗口动态对齐与追随（Window Anchoring）**：
    * **句柄监听**：系统需定时（如每 500ms）监听腾讯麻将窗口的句柄（HWND）及其在桌面上的绝对坐标 `(X, Y)` 与分辨率 `(Width, Height)`。
    * **坐标同步**：一旦游戏窗口发生位移或缩放，悬浮窗必须秒级同步调整自身的位置与大小，确保与游戏画面「像素级」对齐。

### 5.3 UI 渲染与视觉交互需求
悬浮窗的视觉呈现分为三大核心区域：**手牌推荐层**、**副露/动作决策层**、**全局局势面板**。

#### 5.3.1 手牌推荐层（动态锚定渲染）
* **触发时机**：当轮到玩家出牌，且手牌数据为 14 张时触发。
* **视觉组件 1：最优切牌箭头**
    * **样式**：在 AI 算出的最优切牌正上方，渲染一个半透明的**绿色向上箭头（或呼吸灯光圈）**。
    * **位置动态计算**：结合 YOLO11 输出的该张手牌的动态坐标 `Bounding Box [x1, y1, x2, y2]`，将箭头精准端渲染在 `( (x1+x2)/2, y1 - 20 )` 的位置。
* **视觉组件 2：牌效权重分显示**
    * **样式**：在每张手牌的顶部边缘，以小字体实时标统该张牌的切牌权重分。
    * **颜色矩阵**：
        * 分数 $\ge 80$分：绿色提示（强烈推荐切牌）。
        * $40 \le$ 分数 $< 80$分：黄色/橘色提示（可选路线）。
        * 分数 $< 40$分：不显示或灰色显示（极差牌效）。

#### 5.3.2 副露/动作决策层（大动态弹窗）
* **触发时机**：当对家、上家或下家打出某张牌，且 AI 计算出玩家进行「碰」或「杠」能获得正向期望收益（EV）时触发。
* **视觉组件**：在游戏画面中央偏下（不遮挡视线处），弹出一个醒目的半透明悬浮特效。
    * **【 碰 】提示**：黄色亮光，并标贴收益预期（例：*“碰后听牌概率提升 40%”*）。
    * **【 杠 】提示**：红色亮光，并标贴买马/杠分期望（例：*“暗杠直接得分，推荐率 100%”*）。
    * **【 漏碰/过 】提示**：若有碰牌机会但 AI 判定为了做大牌或保留暗杠应故意放弃，则在游戏自带的「碰」按钮上方渲染一个**灰色禁止符号**。

#### 5.3.3 全局局势面板（Side HUD）
* **位置**：固定渲染在游戏窗口内部的左上角或右上角空闲区域（不遮挡头像、聊天框和剩余牌数）。
* **显示内容**：
    1. **当当前向听数**：明确显示「已听牌（听 X 门，共 Y 张）」或「一向听 / 二向听」。
    2. **目标番型趋势**：动态展示当前手牌走向。例如：*清一色（倾向度 75%）*、*碰碰胡（倾向度 20%）*。
    3. **危险张预警**：根据其余三家打牌历史与 AI 局势修正模型，用红色字体列出当前的「绝对点炮危险张」，提醒玩家切勿打出。

## 6. 异常处理与防封安全需求

* **反截图对抗**：悬浮窗在 Windows 层不可使用标准的 GDI+ 直接绘制在屏幕最前端，防止游戏内置的反作弊系统（ACE）通过「后台截取屏幕全图」的方式抓取到算牌器的绿色箭头外框。悬浮窗必须支持**一键开启/关闭直播隐藏模式**（应用 `SetWindowDisplayAffinity` 属性，使常规截屏/录屏软件无法捕捉到 Overlay 图层）。
* **游戏崩溃/关闭处理**：一旦检测到腾讯麻将游戏进程结束或窗口关闭，悬浮窗必须在 **100ms 内自动隐藏或销毁**，防止一个空透明窗口残留在线用户桌面上。
